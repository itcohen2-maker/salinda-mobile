# סלינדה (Salinda) — משחק קלפי חשבון

סלינדה הוא משחק קלפי חשבון מרובה משתתפים שנועד למטרות חינוכיות ופנאי. השחקנים משתמשים בקלפי מספרים ופעולות חשבון כדי לפתור משוואות המבוססות על הטלות קוביות.

## סקירת פרויקט

- **טכנולוגיות ליבה:**
  - **Frontend:** React Native / Expo (SDK 54), TypeScript, Socket.IO Client, Supabase Client.
  - **Backend:** Node.js, Express, Socket.IO, Supabase.
  - **ניהול מדינה (State):** React Context (GameContext), וקובץ `index.tsx` מרכזי (Monolith).
  - **עיצוב:** React Native Stylesheets, גופן Fredoka, תמיכה מלאה ב-RTL (עברית כברירת מחדל).
  - **בדיקות:** Jest (יחידה), Playwright (E2E).

## מבנה התיקיות

- `index.tsx`: נקודת הכניסה הראשית ומרכז הלוגיקה והממשק של המשחק.
- `src/`: קוד אפליקציה מודולרי (מסכים, רכיבים, הוקס, קונטקסט).
- `server/`: קוד המקור של שרת ה-Socket.IO למשחק מרובה משתתפים.
- `shared/`: לוגיקה וטיפוסים (Types) משותפים ללקוח ולשרת (אימות קלפים, i18n, טיפוסים).
- `assets/`: מיתוג, צלילים ונכסים ויזואליים.
- `scripts/`: סקריפטים למחקר וסימולציה של המשחק.
- `docs/`: תיעוד טכני, דוחות מחקר וחוקי המשחק.

## בנייה והרצה

### דרישות מוקדמות
- Node.js (ראה גרסה בקובץ `.nvmrc`).
- אפליקציית Expo Go (לבדיקה במכשירים ניידים).

### פיתוח מקומי
1. **התקנת תלויות:**
   ```bash
   npm install
   npm install --prefix server
   ```
2. **הגדרת סביבה:**
   - העתק את `.env.example` ל-`.env`.
   - הגדר את `EXPO_PUBLIC_LOCAL_SOCKET_SERVER=1` לבדיקת משחק מרובה משתתפים מקומי.
3. **הרצת שרת המשחק:**
   ```bash
   npm run server:dev
   ```
4. **הרצת לקוח ה-Expo:**
   ```bash
   npx expo start
   # הקש 'w' לדפדפן, או סרוק את קוד ה-QR עבור iOS/Android
   ```

### בדיקות
- **בדיקות יחידה:** `npm test`
- **בדיקות קצה (Web):** `npm run test:e2e`

## מוסכמות פיתוח

- **בינאום (i18n):** כל המחרוזות הנראות למשתמש חייבות להתווסף ל-`shared/i18n/{en,he}.ts`. אין להשתמש במחרוזות קבועות (Hardcoded) ברכיבים.
- **תמיכה ב-RTL:** עברית היא השפה העיקרית; יש לוודא שהפריסות (Layouts) מטפלות במימין לשמאל בצורה נכונה.
- **טיפוסים משותפים:** כל שינוי בפרוטוקול התקשורת חייב להתעדכן ב-`shared/types.ts`.
- **לוגיקת משחק:** לוגיקת האימות נמצאת ב-`shared/cardValidation.ts` כדי להבטיח עקביות בין ממשק המשתמש לסמכות השרת.
- **מרובה משתתפים:** יש להשתמש ב-Hook בשם `useMultiplayer` כדי לתקשר עם שרת המשחק.
- **התאמה למשחק האמיתי (חוק):** כל הדגמה/הדגשה בחדר הזהב (ובכל עבודת UI) חייבת להיות מושווית ומותאמת ל**משחק האמיתי** — האלמנטים, המיקומים והמראה כפי שהם בלוח בפועל. אסור להשתמש במיקומים משוערים/placeholder (שברי מסך מנחשים), ואסור לקחת מידות מצילום מסך. **את כל המידות לוקחים מהמשחק עצמו** — מיקומים נמדדים בזמן ריצה (`onLayout`/`measureInWindow`), בדיוק כמו מנגנון `tutorialBus.setLayout`/`HighlightOverlay` של ההדרכה הקיימת.
- **שימוש חוזר במנוע ההדרכה:** חדר הזהב יבנה על מנגנון התרגול החי הקיים (`InteractiveTutorialScreen` + `lessons/` + `tutorialBus` + `HighlightOverlay`), ולא על Spotlight משוער נפרד.
- **מובייל קודם — iOS + Android (חוק):** כל שינוי UI חייב להתאים למסך מובייל ולהיבדק שהוא נראה נכון גם ב-iPhone וגם ב-Android (לא נמרח/חורג בדפדפן רחב). יש להגביל רוחב למסגרת מובייל ולהשתמש בפריסה רספונסיבית.

## פריסה (Deployment)
- **שרת:** פרוס ב-Render (ראה `render.yaml`).
- **Web:** פרוס ב-Vercel.
- **מובייל:** נבנה באמצעות EAS (Expo Application Services).
